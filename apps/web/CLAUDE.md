## 1. Frontend Scope

The frontend is a single-page application (SPA) that provides the three interaction modes defined in the PRD: Input Mode (session ingestion and diff review), Query Mode (natural language questions), and Exploration Mode (browsing the world state). It communicates with the backend exclusively over the versioned REST API at `/api/v1/` — all data is remote, all mutations are server-side. The frontend owns no business logic and performs no world state derivation; it renders what the API returns and submits what the user confirms.

```bash

# Start development server (requires backend running or mocked)

# Type check (no emit)
npx tsc --noEmit

# Lint

npx prettier --write src/
```

```
type check → lint → Vitest → vite build
```

Deployment to Vercel is handled by Vercel's GitHub integration, not by CI (D-045).

## 5. Architecture

The frontend is a feature-sliced SPA. Each interaction mode (`input`, `query`, `exploration`) lives in its own feature directory and owns its routes, components, and data-fetching logic. Cross-cutting concerns — auth, active campaign context, shared UI — live in `store/`, `hooks/`, and `components/domain/`.

**Server state** is managed exclusively by TanStack Query. Components never call `fetch` directly — they use hooks from `src/api/` that wrap `useQuery` / `useMutation`. The API layer in `src/api/` contains one file per resource; each file exports typed request/response functions and query-key factories.

**Client state** is handled by focused Zustand stores: one for auth (token, user identity), one for active campaign context, and one for UI flags (active mode, modal state, etc.). Stores do not replicate server data — that belongs in TanStack Query cache.

**Auth** — the access token is stored in memory inside the auth Zustand store (never `localStorage`). Every API request attaches it as `Authorization: Bearer <token>`. On 401, the client attempts a silent refresh via `POST /api/v1/auth/refresh` and retries once. If the refresh fails, the user is redirected to login.

**Routing** is handled by React Router v6. Routes map to interaction modes and their sub-views. All routes behind the campaign context require an authenticated user and an active campaign in store.

**API contract** — TypeScript types in `src/types/` are hand-maintained mirrors of backend DTOs. There is no code generation in v1 (D-030). When a backend DTO changes, update the corresponding type in the same PR.

## 6. Frontend Domain Concepts

### Diff Review Screen (Input Mode)
The core UI of Input Mode. Displayed when a session reaches `draft` status after processing. Opens with a 1–3 sentence AI-generated narrative summary header (D-005). Below that: extraction categories (Actors, Spaces, Events, Relations), each showing only changed fields for existing entities (D-006) and full profiles for new entities (D-007). Each item is a card the user can **Accept**, **Edit inline**, or **Delete**. A single **Commit** button finalizes all changes. The Commit button is disabled until all UNCERTAIN cards are resolved (D-042).

### UNCERTAIN Card
A specific diff card type produced when the AI cannot confidently determine whether an extracted mention maps to an existing entity or is new. The user must explicitly choose **Same entity (match)** or **Different entity (new)** — there is no defer option. A progress indicator shows `N items require resolution before commit`. The backend enforces this with a `422` if any UNCERTAIN entities appear in the commit payload (D-042).

### Conflict Warning Card
A non-blocking card surfaced in the diff when a new extraction contradicts existing world state (D-033). The commit is not blocked. The user must include an acknowledgment for each conflict warning in the commit payload — the backend rejects with `422` if conflicts were detected and the `acknowledged_conflicts` array is absent or empty.

### Session Ingestion Flow
A multi-step asynchronous flow: `POST /sessions` (submit) → poll `GET /sessions/{id}/status` until status leaves `processing` → `GET /sessions/{id}/diff` (retrieve draft) → user review → `POST /sessions/{id}/commit`. The draft diff is stored server-side in `sessions.diff_payload` — if the user closes mid-review, the diff is recoverable on return. At most one session per campaign may be in `processing` or `draft` state simultaneously (D-054).

### Exploration Mode Views
Four read-only views — Timeline, Entities, Spaces, Relations — composing Exploration Mode. No direct world state edits from any view (D-010). Every entity, space, and relation renders a **"Propose a change"** affordance that is visible but inactive in v1 (D-012). Any member can attach annotations (comments) to any entity, space, or relation; annotations are clearly marked as non-canonical commentary (D-011).

### Relations Graph
The Relations view uses React Flow v12 to render a graph of Actor↔Actor and Actor↔Space connections. This is the only view requiring a dedicated graph library. It lives at `features/exploration/relations/`.

### Query Mode
The user submits a free-text question; the frontend posts to `POST /api/v1/campaigns/{id}/queries` and waits synchronously for the response (D-052). No streaming in v1. The response carries an `answer` text and a `citations` array mapping claims to specific session numbers. Citations must be rendered as navigable links. There is no Q&A history panel in v1 — queries are stateless (D-058).

### Annotations
Any campaign member can post a free-text annotation on any Actor, Space, Relation, or Event from Exploration Mode. They are immutable after creation and visually distinguished from canonical world state content. Render as a comment section beneath each entity profile.

### Role-Gated UI
UI affordances are gated by campaign role: Input Mode actions (upload, diff review, commit) are visible only to `gm` and `editor` roles. `player` role sees only Query Mode and Exploration Mode. The `admin` role has full platform access. Role is not embedded in the JWT — it is resolved server-side, so the client must derive the role from the campaign membership response, not from the token.

## 7. Frontend Conventions

### RULE: Never edit `components/ui/`
`src/components/ui/` is auto-generated by the shadcn/ui CLI. Any change will be overwritten on the next shadcn update. Create wrapper or composition components in `src/components/domain/` instead.

**Rationale:** D-030 — shadcn/ui is the component library. Its primitives are owned by the tool.

### RULE: Feature-scoped files
Components, hooks, and API calls that belong to a single feature live inside that feature's directory. Nothing moves to `components/domain/` or `hooks/` until it is used by two or more features.

### RULE: Server state lives in TanStack Query; client state lives in Zustand
Never put server-fetched data in Zustand. Never fetch data outside of TanStack Query hooks. The boundary is: if the data comes from the API, it belongs in the TanStack Query cache.

**Rationale:** ARCHITECTURE.md §4.3 — the two-tool split is explicit and intentional.

### RULE: Zustand stores are small and focused
One store per distinct concern. There is no global mega-store. Defined concerns: auth context, active campaign context, UI flags. Adding a new store requires justifying why it cannot belong to an existing one.

### RULE: TypeScript types are hand-written and co-located with the API client
Types in `src/types/` mirror backend DTOs. When a backend DTO changes, the corresponding type in `src/types/` must be updated in the same PR. The TypeScript compiler is the primary drift detector. There is no code generation in v1.

**Rationale:** ARCHITECTURE.md §4.4 — explicitly documented, upgrade path to OpenAPI generation deferred to v2.

### RULE: All IDs are UUIDs; all timestamps are ISO 8601 UTC
Do not convert IDs. Do not store timestamps in local formats in state — parse to display only at the render boundary.

### RULE: The Commit button is always controlled
The Commit button in diff review is a controlled component. Its `disabled` state is derived from the count of unresolved UNCERTAIN cards. Any change to the diff card resolution state must flow through a state variable that the Commit button reads.

**Rationale:** D-042 — UNCERTAIN resolution is mandatory before commit. The UI must enforce this independently of the backend's `422` validation (which is defence in depth, not the primary guard).

### RULE: No localStorage for auth tokens
The JWT access token is stored in memory (Zustand auth store) only. It is never written to `localStorage` or `sessionStorage`. Refresh tokens are `httpOnly` cookies managed by the browser — the frontend never reads them.

### RULE: Accessibility testing is not optional
Every component in `components/domain/` and every feature-level flow must have an axe-core test via `vitest-axe`. Diff review cards, entity profiles, and the relations graph are explicitly called out in ARCHITECTURE.md §9.6. Add axe assertions alongside every new React Testing Library test.

### RULE: Exploration Mode is read-only
No form, button, or input in any Exploration Mode view should mutate world state. The only write actions in Exploration Mode are: posting annotations (D-011) and submitting proposals (affordance visible, inactive in v1 — D-012).

### RULE: Styling uses Tailwind utility classes via shadcn/ui
Do not write custom CSS unless absolutely unavoidable. Use Tailwind classes. Theme tokens are defined through shadcn/ui's configuration.

## 8. Key Frontend Files

- **`src/main.tsx`** — app entry point; sets up the React root, router, and TanStack Query client provider.
- **`src/api/`** — the typed HTTP client layer; every API resource has one file here; this is where all `useQuery` / `useMutation` hooks are defined and re-exported to features.
- **`src/types/`** — TypeScript interfaces mirroring backend DTOs; these must stay in sync with backend DTO changes manually.
- **`src/store/`** — Zustand store definitions; the auth store manages the in-memory access token and current user; the campaign store manages the active campaign context.
- **`src/features/input/`** — contains the session submission form, the status polling logic, and the diff review screen with its UNCERTAIN and conflict card components.
- **`src/features/exploration/relations/`** — the React Flow graph; the most complex single component in the app; keep its dependencies isolated.
- **`src/components/ui/`** — shadcn/ui auto-generated primitives; read-only.
- **`src/components/domain/`** — shared domain-aware components used across two or more features (e.g., `EntityCard`, `AnnotationThread`, `SessionBadge`).
- **`src/hooks/`** — shared hooks not tied to a single feature (e.g., `useCurrentCampaignRole`, `useTokenRefresh`).

## 9. Common Frontend Workflows

### Adding a new page / route

1. Determine which feature the page belongs to (`input`, `query`, `exploration`, or a new feature directory).
2. Create a new component file inside `src/features/<feature>/`.
3. Add the route to the router configuration in `src/main.tsx` (or wherever the route config is centralized).
4. If the route requires authentication or a specific role, wrap it with the appropriate route guard component.
5. Add a navigation entry if the page should appear in the sidebar or mode switcher.

### Adding a new component

1. If the component belongs to a single feature, create it in `src/features/<feature>/`.
2. If it is used across two or more features, create it in `src/components/domain/`.
3. Never create it in `src/components/ui/` — that directory is owned by shadcn/ui.
4. Export the component from the feature's or directory's index file.
5. Write a Vitest + React Testing Library test covering the component's primary interactions.
6. Add an axe-core assertion to the test if the component is interactive or is part of a critical user flow.

### Connecting a component to the backend API

1. Identify the backend endpoint in ARCHITECTURE.md §7 or the root `CLAUDE.md` §5.
2. Add or extend the corresponding file in `src/api/` with the typed request/response function.
3. Add a query key factory if the call is a `GET` (used by TanStack Query's cache invalidation).
4. Create a `useQuery` or `useMutation` hook in the same `src/api/` file.
5. Import and call the hook from the feature component.
6. Handle loading, error, and empty states explicitly — never leave them implicit.

### Adding a new form with validation

1. Identify which feature the form belongs to and create the component there.
2. Use **React Hook Form** with shadcn/ui's `Form`, `FormField`, `FormItem`, `FormLabel`, `FormControl`, and `FormMessage` primitives. This stack is designed to compose directly — do not reach for a different form approach (D-067).
3. Validate on the client before submission; also handle `400` validation errors from the API and map error `field` values from the response envelope to form field errors using `setError` from `useForm`.
4. On successful mutation, invalidate the relevant TanStack Query cache key.
5. Disable the submit button while the mutation is in-flight.

### Writing a frontend test

1. Create a test file co-located with the component or hook: `MyComponent.test.tsx`.
2. Use Vitest as the test runner and React Testing Library for component rendering.
3. Assert on visible output and user interactions — not on implementation details like state variables.
4. For any interactive component, add an axe-core check: `expect(await axe(container)).toHaveNoViolations()`.
5. For Zustand stores and custom hooks, test them in isolation using Vitest with no component rendering.
6. For API client hooks, mock the `fetch` layer — do not make real network calls in tests.

## 10. Frontend-Specific Gotchas

**`components/ui/` is a write-once zone.** Running `npx shadcn add <component>` overwrites files in that directory. If you've customized anything there, it will be lost silently. Always wrap in `components/domain/`.

**Campaign role is not in the JWT.** The JWT carries only `user_id` and `is_admin`. Campaign-level role (`gm`, `editor`, `player`) must be read from the campaign membership API response (D-043). If a component gates a feature on role, it must read from the campaign context store — not parse the token.

**One draft per campaign.** `POST /api/v1/campaigns/{id}/sessions` returns `409` if another session is already in `processing` or `draft` state (D-054). The submission UI must check for an existing draft and surface a recovery path (retrieve the existing draft via `GET /sessions/{id}/diff`) before allowing a new submission.

**UNCERTAIN cards block commit — backend also enforces this.** The `422` from the commit endpoint is defence in depth; the primary guard is the UI. If a user somehow submits with unresolved UNCERTAIN cards, the backend rejects. Always treat `422 UNCERTAIN_ENTITIES_PRESENT` as a UI bug to fix, not just an error to handle.

**Conflict acknowledgments are required when conflicts exist.** The commit payload requires a non-empty `acknowledged_conflicts` array if the diff contained conflict warnings. An empty array `[]` is valid only when no conflicts were detected. Sending `[]` when conflicts exist triggers a `422`. The commit builder must collect conflict acknowledgments separately from card-level accept/edit/delete actions.

**Token refresh is transparent.** On a `401`, the client retries once after refreshing. If that retry also returns `401`, redirect to login. Do not surface `401` errors to the user directly — they are a session management concern, not a user error.

**Query Mode is synchronous with a hard timeout.** The server holds the connection and returns the complete response. The client must handle `504 QUERY_TIMEOUT` as a user-facing message ("The query timed out — try rephrasing or narrowing the question"), not an unhandled error. No streaming or partial responses in v1 (D-052).

**`POST /sessions` response does not include the diff.** It returns a `session_id` and `status: processing`. The client must poll `GET /sessions/{id}/status` until `status` is `draft` or `failed` before attempting to retrieve the diff. Polling on a `failed` status should surface the `failure_reason` and `message` from the status response.

**Summary too large is a `400`, not a `422`.** If a submitted summary exceeds the token limit, the API returns `400` with error code `SUMMARY_TOO_LARGE` and a `max_tokens` field. Surface this as a user-facing message with the limit and the suggestion to split the summary into multiple sessions (ARCHITECTURE.md §6.3).

**Timeline pagination is keyset, entity lists are offset.** These require different data-fetching patterns. Timeline (`GET /campaigns/{id}/timeline`) uses keyset cursors; actor/space/event/relation list endpoints use offset pagination (D-055). Do not apply the same pagination strategy to both.

**The "Propose a change" affordance is a stub in v1.** It must render on every entity, space, and relation in Exploration Mode. Clicking it in v1 does nothing meaningful — the approval pipeline ships in v2 (D-012). Render the affordance as a visible but disabled/placeholder UI element with a tooltip indicating the feature is coming.

**Annotation immutability.** Annotations have no `updated_at` and cannot be edited. Once posted, the only write operation is deletion (by the author, or by the GM). Do not render an edit button on annotation cards.

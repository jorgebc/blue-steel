# Module Name: Query Mode

## 1. Overview

Query Mode lets any campaign member ask free-form questions about the campaign ("Who betrayed the party in the mines?") and receive an answer generated by the LLM but **grounded exclusively in committed world state**, with citations pointing to the sessions that support each claim (D-003 — citations prevent hallucinated lore from being taken as canon). Execution is synchronous and stateless: one question, one answer, no streaming and no stored history (D-052, D-058) — navigating away discards the answer.

Because every query costs real LLM money, the module is wrapped in layered cost governance (D-096): per-user rate limiting, a shared daily spend cap, a hard time deadline, token budgets, and a bounded executor pool.

## 2. Capabilities & Use Cases

- **Use Case / Action:** Ask a natural-language question — ✅ Implemented
- **Actor:** Any campaign member (GM, Editor, Player)
- **Functional Description:** The user types a question (max 2000 characters, live counter). The backend embeds the question, retrieves the top-N (default 8) most semantically similar committed entity versions via pgvector, assembles a context prompt (with prompt-injection hardening), and makes a single LLM call instructed to answer only from that context and cite supporting sessions. The response is `{ answer, citations[] }`; citations returned by the LLM are validated against the actually retrieved context before being shown.
- **Technical Reference / Source Files:** `POST /api/v1/campaigns/{id}/queries` — `apps/api/src/main/java/com/bluesteel/adapters/in/web/query/QueryController.java`, `apps/api/src/main/java/com/bluesteel/application/service/query/QueryService.java`, retrieval adapter `apps/api/src/main/java/com/bluesteel/adapters/out/persistence/embedding/EntityQueryRetrievalAdapter.java`, answering adapter `apps/api/src/main/java/com/bluesteel/adapters/out/ai/SpringAiQueryAnsweringAdapter.java` (+ `QueryResponseParser.java`); UI: `apps/web/src/features/query/QueryPage.tsx`, `components/QuestionForm.tsx`

- **Use Case / Action:** Enforce per-user rate limiting — ✅ Implemented
- **Actor:** System
- **Functional Description:** A sliding-window limiter per (user, campaign) tuple — default 10 requests per 60 seconds — returns 429 when exceeded; the key map is bounded to prevent memory growth. The UI shows a non-fatal warning banner suggesting a short wait.
- **Technical Reference / Source Files:** `apps/api/src/main/java/com/bluesteel/adapters/in/web/query/QueryRateLimiter.java`, error handling in `QueryPage.tsx`

---

- **Use Case / Action:** Enforce the shared daily cost cap — ✅ Implemented
- **Actor:** System
- **Functional Description:** All LLM spend (all pipeline stages plus queries) is accumulated per UTC day in memory; once the configured cap (default $1.00/day) is reached, further queries return 503 `QUERY_COST_CAP` until midnight UTC. A provider-level hard cap in the Gemini console backs this up. The UI explains that the daily limit was reached.
- **Technical Reference / Source Files:** `apps/api/src/main/java/com/bluesteel/adapters/out/ai/InMemoryDailyCostAccountingAdapter.java`, enforcement in `QueryService.java`

---

- **Use Case / Action:** Enforce deadline, token budget, and concurrency bounds — ✅ Implemented
- **Actor:** System
- **Functional Description:** Queries run on a dedicated bounded executor (default 2 threads, queue of 20) under a synchronous deadline (default 20 s → 504 `QUERY_TIMEOUT`, UI suggests rephrasing/narrowing) and a per-request token budget (default 6000 tokens → 422). LLM faults are mapped to specific error codes rather than generic 500s.
- **Technical Reference / Source Files:** `QueryService.java`, executor configuration in the backend config classes, `apps/api/src/main/java/com/bluesteel/adapters/in/web/GlobalExceptionHandler.java`

---

- **Use Case / Action:** Check remaining query budget — ✅ Implemented
- **Actor:** Any campaign member
- **Functional Description:** A read-only usage endpoint (does not consume the rate limit) reports the shared daily budget consumption and the caller's remaining requests in the current window. The UI displays this beside the question form and refreshes it after each submission.
- **Technical Reference / Source Files:** `GET /api/v1/campaigns/{id}/queries/usage` — `QueryController.java`, `apps/api/src/main/java/com/bluesteel/application/service/query/QueryUsageService.java`, `apps/web/src/features/query/components/QueryUsageNotice.tsx`

## 3. Core User Journeys (Workflows)

**Journey: Ask → verify**
1. Member opens `/campaigns/:id/query` from the sidebar.
2. Types a question (counter warns near the 2000-char limit) and submits; the form disables and a skeleton holds the answer area (no spinners by design).
3. The grounded answer appears with a citation list; the usage notice updates the remaining budget.
4. The member clicks a citation → lands on the read-only session detail page → verifies the claim against the original session text → navigates back and asks a follow-up (the previous answer is gone — stateless by design).

**Journey: Hitting a governance limit**
1. A member fires several questions in quick succession → 429; a warning banner asks them to wait a few seconds. The window resets and they continue.
2. Later that day the table collectively exhausts the daily cap → 503; the banner explains the daily limit is reached and querying resumes at UTC midnight.
3. A very broad question exceeds the 20-second deadline → 504; the banner suggests rephrasing or narrowing the scope.
